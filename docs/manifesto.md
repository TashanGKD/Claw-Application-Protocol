# CAP Manifesto

## The Claim

> People should act like bosses, not get trapped in the anxiety of endless configuration.  
> Users should stay at the level of intent. Systems should absorb the churn below.

This is the practical claim behind **Claw Application Protocol (CAP)**.

AI systems are producing more models, tools, runtimes, integrations, agent loops, and workflow patterns at a pace most users cannot reasonably track.
If every new layer requires the user to learn a new operational model, the system becomes more powerful in theory and more exhausting in practice.

That is the wrong direction.

## The Shift

Software is moving from feature execution toward task execution.

The previous generation asked:

> How can one system call another capability?

That question produced APIs, workflow tools, integration layers, and later AI-enhanced products.

The more important question now is:

> How can a system keep a real task moving forward over time?

This is a different problem.
It needs a different center.

## The Wrong Default

The current AI ecosystem often pushes users toward the wrong abstraction.

Users are asked to think about:

- which model to use
- which tool to connect
- which server to expose
- which prompt pattern to maintain
- which workflow to rebuild after the next ecosystem shift

That is operational burden disguised as empowerment.

For most people, the stable interaction should remain much simpler:

- state the goal
- correct the direction
- approve or reject outcomes
- carry long-term preferences

Everything below that layer should be handled by the system.

## The New Center

CAP proposes that the first-class object of an AI-native application should be the **task object**.

A task object is not just a prompt, ticket, or chat turn.
It is a runtime entity with:

- a goal
- a state
- a history
- constraints
- artifacts
- pending questions
- review outcomes
- links to user-specific standards

This is what allows a system to do more than respond once.
It allows the system to remain inside the work.

## Capability Access Is Not Task Continuity

CAP does not compete with lower layers.
It depends on them.

- APIs expose functions
- MCP exposes capabilities to models and agents
- skills package reusable task fragments

All of these remain useful.
But none of them, by themselves, defines the runtime of a living task.

A system can have excellent tools and still fail real work if it cannot preserve:

- task state
- user expectations
- revision loops
- execution continuity across time

The important shift is from **capability access** to **task continuity**.

## Personal Standards Matter

In a persistent assistant environment, not all users mean the same thing by "done".

A strong system should know:

- what evidence bar the user expects
- how they prefer work to be structured
- what shortcuts they reject
- what quality issues they repeatedly correct
- what kinds of outputs they consider usable

CAP treats this not as optional personalization, but as part of the runtime context for real task completion.

## What CAP Is Trying To Make Possible

CAP begins as an open initiative and minimal protocol draft for:

- modeling task-native AI applications
- defining their core objects
- defining a minimal runtime lifecycle
- keeping a stable task-facing user entry point while execution changes underneath
- describing how capabilities fit inside task execution without becoming the product center

The goal is not to over-standardize early.
The goal is to establish a shared language for a software shape that is becoming necessary.

## What Success Looks Like

CAP succeeds if it helps builders create systems where:

- the task is first-class
- state survives across sessions
- users stay at the level of intent rather than tool configuration
- actions are selected in service of the task, not isolated prompts
- outputs are reviewed against user-specific standards
- work can pause, resume, and continue without losing coherence

## The Boundary

The next important boundary in AI software is not only tool use.
It is whether the system can continuously run around a task object while keeping the user interface stable and the execution layer adaptable.

That is the layer CAP is trying to name, clarify, and make buildable.
